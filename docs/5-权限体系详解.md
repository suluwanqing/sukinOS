# 权限体系详解

SukinOS 的权限认证是一个**五层递进**的体系，从全局中间件到单个路由处理函数，层层加固。

## 一、权限体系总览

SukinOS 的权限认证是一个**五层递进**的体系，从全局中间件到单个路由处理函数，层层加固。

### 核心概念：三层权限控制

真正的请求拦截逻辑在**服务端**由三层独立机制协同完成，不是单层防御：

| 层级 | 实现 | 作用范围 | 检查依据 | 拦截结果 |
|---|---|---|---|---|
| **L1 - 配置驱动层** | `RoutePermissionMiddleware` 中间件 | 所有路由 | `D_SystemConfig` 的路由权限配置 | 403 权限不足 |
| **L2 - 代码声明层** | `@RequireRoot` / `@RequirePermission` / `@RequireRole` 装饰器 | 加了装饰器的特定路由 | `user.root`、`user.permission.role`、`user.permission.keys` | 403 权限不足 |
| **L3 - 身份认证层** | `Depends(verify_auth())` 依赖注入 | 路由参数级 | JWT Cookie（Access Token + Refresh Token） | 401 未登录 |

**三者的关系不是互斥，而是递进加固**：

```
请求 → L1 中间件检查路由权限配置
         ├─ 根据 allowed_roles / allowed_users 判断
         ├─ 非 root 且不匹配 → ❌ 拦截（403），不进路由
         └─ 匹配 → ✅ 放行进入路由
                  ├─ L3 Depends(verify_auth())
                  │   → 解析 JWT → 验证身份 → 获取 User 对象
                  │   → Token 过期 → 自动 Refresh Token 静默刷新
                  │
                  ├─ L2 @RequireRoot()
                  │   → 检查 user.root
                  │   → 非 root → ❌ 拦截（403）
                  │
                  ▼ 执行实际业务逻辑
```

### 为什么需要三层？

| 层 | 解决的问题 | 如果缺少会怎样 |
|---|---|---|
| L1 配置驱动 | 全局兜底，管理员可通过面板动态调整 | 新人忘加装饰器，接口直接裸奔 |
| L2 代码声明 | 代码显式标注接口的权限意图 | 配置被人误删/改错，接口变公开 |
| L3 身份认证 | JWT 验证 + 静默刷新，获取当前用户 | 接口里拿不到 `current_user`，也做不了身份校验 |

> **配置驱动兜底 + 代码声明加固** = 互相冗余、互相 backup。即使管理员在面板误操作将某管理路由改成了公开，代码中的 `@RequireRoot()` 和 `Depends(verify_auth())` 依然能拦截。

### 五层总览

```
┌─────────────────────────────────────────────────────┐
│  Layer 5: 前端 RoutePermission 中间件                │
│  (axios 请求拦截器，本地拦截越权请求，避免无效网络调用) │
├─────────────────────────────────────────────────────┤
│  Layer 4: StaticAuthMiddleware (静态资源鉴权)         │
│  (APP 文件下载、用户头像访问，资源所有者 + 权限池校验)   │
├─────────────────────────────────────────────────────┤
│  Layer 3: Depends(verify_auth()) - 身份认证层         │
│  (JWT 验证 + Refresh Token 静默刷新，获取当前用户)     │
├─────────────────────────────────────────────────────┤
│  Layer 2: @RequireRoot / @RequirePermission - 代码层 │
│  (装饰器显式声明接口权限意图，二次加固)                 │
├─────────────────────────────────────────────────────┤
│  Layer 1: RoutePermissionMiddleware - 配置驱动层      │
│  (基于 D_SystemConfig 动态配置，全局兜底)              │
└─────────────────────────────────────────────────────┘
```

## 二、L1 - 全局路由权限中间件

**文件**: `Middleware/routePermission.py`

### 2.1 流程

```
请求到达
  │
  ├─ 白名单路径？ → 放行
  │   publicPaths: 精确匹配
  │   publicPrefixes: /static, /resource, /user/status/
  │   selfPrefixes: /system/permission/ (自身权限管理)
  │
  ├─ 已登录用户？
  │   ├─ root → 放行
  │   └─ check_route_permission()
  │       ├─ 路由无配置 → 放行（默认仅要求登录）
  │       ├─ require_auth=false → 放行
  │       ├─ _auto=true → 放行（自动注册未配置）
  │       ├─ allowed_roles 匹配 → 放行
  │       ├─ allowed_users 匹配 → 放行
  │       └─ 否则 → 403 权限不足
  │
  └─ 未登录用户？
      ├─ require_auth=true（默认）→ 401 请先登录
      └─ require_auth=false → 放行
```

### 2.2 配置结构

存储在 `D_SystemConfig` 表的 `system_api_route_permission` 键下：

```json
{
  "routes": {
    "/system/usermanage/users": {
      "require_auth": true,
      "allowed_roles": ["root"],
      "allowed_users": [1, 2, 3],
      "methods": ["GET", "POST", "PUT", "DELETE"],
      "description": "用户管理接口",
      "_locked": true,
      "_auto": false
    },
    "/user/status/token": {
      "require_auth": false,
      "_auto": false,
      "_locked": false
    }
  }
}
```

### 2.3 关键字段说明

| 字段 | 类型 | 说明 |
|---|---|---|
| `require_auth` | bool | 是否需要认证（旧兼容: `requireAuth`） |
| `allowed_roles` | string[] | 允许访问的角色列表 |
| `allowed_users` | int[] | 允许访问的用户 ID 列表 |
| `_auto` | bool | 系统自动扫描注册（非手动配置） |
| `_locked` | bool | 系统受保护路由（禁止修改） |

### 2.4 自动扫描注册

应用启动时（`exit.py:auto_seed_route_permissions`）扫描所有 `app.routes`：
- **受保护前缀** (`/system/permission/`) → 标记 `_locked: true`，仅 root
- **公开前缀** (`/resource/`, `/user/status/`) → `require_auth: false`
- **其他路由** → `_auto: true`，允许管理员配置

### 2.5 `_auto` 机制详解

`_auto` 是一个**标记位**，表示"这条路由是自动扫描注册的，管理员还没手动配置过它的权限规则"。

**启动时自动注册**：

```python
# exit.py — 所有新路由自动带 _auto: true
entry = {
    "require_auth": True,
    "allowed_roles": [],
    "allowed_users": [],
    "_auto": True,   # ← 自动注册标记
}
```

**中间件遇到 `_auto: true` 直接放行（不检查 allowed_roles/allowed_users）**：

```python
# Api/Common/permission.py — check_route_permission()
def check_route_permission(db, path, method, user):
    rule = routes.get(path)
    if not rule:
        return True               # 不在配置中 → 放行
    if rule.get("_auto"):
        return True               # ← 自动注册未配置 → 放行，不拦截
    # 下面是管理员手动配置后的严格检查
    if user.role in rule.get("allowed_roles", []):
        return True
    if user.id in rule.get("allowed_users", []):
        return True
    return False                  # 拒绝
```

**管理员手动配置后 `_auto` 被移除，开始严格检查**：

```python
# Api/Location/appPermission.py — 更新路由权限时
def _normalize_route_payload(payload):
    payload.pop("_auto", None)   # ← 管理员一配置，_auto 就被删除
    payload.pop("_locked", None)
    return payload
```

**`_auto` 的完整生命周期**：

| 阶段 | `_auto` 值 | 中间件行为 | 意图 |
|---|---|---|---|
| 刚开发完新接口，启动运行 | `true` | 放行，不检查 | 避免新接口未配置权限 → 全片 403 |
| 管理员通过前端配置了该路由的允许角色/用户 | 被移除 | 严格按 `allowed_roles` / `allowed_users` 检查 | 一旦有人管了，就严格管起来 |
| 启动后新路由自动加入 | `true` | 放行 | 给管理员缓冲期去配置 |

> **设计动机**：这样新加一个 API 接口，不会因为还没配权限就立即 403 报错。管理员有充分时间去前端面板配置规则。配置完成后自动转入严格模式，中间没有间隙。
- 已有配置不会被覆盖

## 三、L2 - 静态资源鉴权

**文件**: `Middleware/staticAuth.py`

### 3.1 处理器映射

```python
AUTH_HANDLERS = {
    APP_FILE_MOUNT: handle_sukinos_app_auth,
    RESOURCE_USER_MOUNT_PATH: handle_public_auth,
}
```

### 3.2 APP 文件鉴权 (`handle_sukinos_app_auth`)

```
请求下载资源
  │
  ├─ 资源不存在 → 404
  │
  ├─ 非私有(is_private=false) → 放行
  │
  └─ 私有(is_private=true)
      ├─ JWT 认证
      │   ├─ access_token 有效 → 提取用户
      │   └─ access_token 过期 → refresh_token 静默刷新
      │       └─ 成功 → 注入新 Cookie
      │
      ├─ 资源拥有者(user_id匹配) → 放行
      │
      ├─ 权限注册池(registry_enabled=true)
      │   └─ check_permission() 通过 → 放行
      │
      └─ 否则 → 403 无权限
```

## 四、L3 - FastAPI 依赖注入

### 4.1 `verify_auth()` 工厂函数

**文件**: `Api/Common/Auth.py`

支持多种认证模式：

```python
# 仅登录验证（最常用）
Depends(verify_auth())

# 要求 root
Depends(verify_auth(mode="superuser"))

# 角色校验
Depends(verify_auth(mode="role", required_role="admin"))

# 权限位校验
Depends(verify_auth(mode="power", power_type="first", required_bits=[0x01]))
```

### 4.2 `require_root()` 依赖

```python
Depends(require_root())
# 等价于 verify_auth(mode="superuser")
```

### 4.3 `require_permission()` 依赖

```python
Depends(require_permission())
# 根据路由权限配置检查用户是否有权访问
```

### 4.4 `verify_refresh_token_dep` 依赖

专门验证 Refresh Token，用于刷新接口。

## 五、L4 - 装饰器

### 5.1 `@RequireRoot()`

**文件**: `Api/Common/permission.py`

```python
@router.get("/admin")
@RequireRoot()
async def handler(current_user=Depends(verify_auth())):
    ...
```

- 从 `kwargs` 获取 `current_user`
- 检查 `user.root` 属性
- root 放行，非 root 返回 403

### 5.2 `@RequirePermission(permission_key)`

```python
@router.get("/users")
@RequirePermission("system:users:view")
async def handler(current_user=Depends(verify_auth())):
    ...
```

- 检查 `user.permission.keys` 列表中是否包含指定 key
- root 用户无条件放行

### 5.3 `@RequireRole(role)`

```python
@router.get("/developer")
@RequireRole("developer")
async def handler(current_user=Depends(verify_auth())):
    ...
```

- 检查 `user.permission.role` 是否匹配
- root 用户无条件放行

## 六、L5 - 前端路由权限中间件

**文件**: `src/middleware/routePermission/`

### 6.1 架构设计

```
RoutePermission (React 组件, 渲染 null)
  │ 启动时加载配置 → updateRoutePermissionCache()
  │ 监听 userInfo → updateCurrentUser()
  ▼
cache.js (模块级缓存)
  │ checkRoutePermission(method, path): boolean
  ▼
Axios 请求拦截器
  │ 发送前调用 checkRoutePermission()
  │ 越权请求直接 return 不走网络
```

### 6.2 缓存逻辑

```javascript
checkRoutePermission(method, path) {
  if (!configLoaded) return true     // 配置未加载 → 放行
  if (!currentUser) return false     // 未登录 → 拦截
  if (currentUser.root) return true  // root → 放行
  if (!routeConfig?.routes) return true

  const rule = routeConfig.routes[path]
  if (!rule) return true             // 无配置 → 放行
  if (rule.require_auth === false) return true

  // 检查 allowed_roles / allowed_users
  const role = currentUser.permission?.role
  if (allowedRoles.includes(role)) return true
  if (allowedUsers.includes(userId)) return true

  return false  // 拦截
}
```

## 七、APP 权限体系

### 7.1 权限注册池

**文件**: `Func/system/permissionManager.py`

`D_AppPermissionRegistry` 表管理第三方 APP 的权限控制：

```json
{
  "resource_id": "app-abc-123",
  "permission_enabled": true,
  "actor_rules": {
```

**检查优先级**: denied_users > denied_roles > allowed_users > allowed_roles

### 7.2 系统 APP 可见性

**文件**: `Func/system/permissionManager.py:check_system_app_access()`

```
用户访问系统 APP
  ├─ root → 放行
  ├─ APP 隐藏(hidden=true) → 拒绝
  ├─ allowed_users 匹配 → 放行
  ├─ allowed_roles 匹配 → 放行
  ├─ default_visible_to 匹配 → 放行
  └─ 否则 → 拒绝
```

## 九、三层权限控制协同详解

### 9.1 三层关系总览

SukinOS 的请求拦截逻辑在服务端由 **三层独立机制** 协同完成，不是单层防御：

| 层级 | 实现 | 作用范围 | 检查依据 | 拦截结果 |
|---|---|---|---|---|
| **L1 - 配置驱动层** | `RoutePermissionMiddleware` 中间件 | 所有路由 | `D_SystemConfig` 的 `system_api_route_permission` 配置 | `403` 权限不足 |
| **L2 - 代码声明层** | `@RequireRoot` / `@RequirePermission` / `@RequireRole` | 加了装饰器的特定路由 | `user.root`、`user.permission.role`、`user.permission.keys` | `403` 权限不足 |
| **L3 - 身份认证&JWT静默刷新层** | `Depends(verify_auth())` / `Depends(require_root())` | 路由参数级 | JWT Access Token / Refresh Token Cookie | `401` 未登录 |

### 9.2 实际请求执行链路

以一个典型的 root 管理接口为例（如系统概览统计）：

```python
@router.get("/overview")
@RequireRoot()                              # ← L2 装饰器
async def get_system_overview(
    current_user: User = Depends(verify_auth()),  # ← L3 依赖注入
    db: Session = Depends(get_db),
):
```

当一个请求进来时，实际的执行链路是：

```
HTTP Request → RoutePermissionMiddleware (L1)
  │
  ├─ step 1: 检查白名单
  │   → /system/usermanage/overview 不在 publicPrefixes 中
  │   → 需要检查权限
  │
  ├─ step 2: 解析用户身份
  │   → 读取 Cookie: access_token="Bearer xxx"
  │   → decode JWT → 获取 account
  │   → 从 DB 查询完整用户对象（含 root、permission、role）
  │   → 用户 role = "user", root = false
  │
  ├─ step 3: 读取路由权限配置
  │   → 从 D_SystemConfig 加载 system_api_route_permission
  │   → 查找 /system/usermanage/overview 的规则
  │   → 查到: { "allowed_roles": ["root"], "_auto": false }
  │       (注意: _auto=false, 说明管理员已手动配置过)
  │
  ├─ step 4: check_route_permission()
  │   → root? 否 → 继续
  │   → _auto? false → 继续（需要严格检查）
  │   → role="user" 在 allowed_roles=["root"] 中？ 否
  │   → user.id 在 allowed_users=[] 中？ 否
  │   → ❌ 返回 False
  │   → 中间件返回 403
  │
  ▼ （如果中间件放行，进入路由函数）
  ├─ Depends(verify_auth()) ← L3
  │   → access_token 有效 → 解析 payload → 查询 User → 返回
  │   → access_token 过期 → 尝试 refresh_token 静默刷新
  │     → refresh_token 有效 → 签发新 Token 对 → 注入 Response Cookie
  │     → refresh_token 过期 → raise 401
  │   → 返回 User 对象（含 root / permission 等信息）
  │   → 将 User 注入 current_user 参数
  │
  ├─ @RequireRoot() ← L2
  │   → 从 kwargs 获取 current_user
  │   → current_user.root = false
  │   → ❌ raise 403
  │
  ▼ （如果全部通过，执行业务逻辑）
  └─ 正常执行 → RequestLogMiddleware 记录日志 → 返回响应
```

### 9.3 为什么需要三层？

| 层 | 解决的问题 | 如果缺少会怎样 |
|---|---|---|
| **L1 配置驱动** | 全局兜底，管理员可**零代码**通过面板调整 | 新人忘加装饰器，接口直接裸奔（测试环境 -> 生产环境风险） |
| **L2 代码声明** | 代码**显式标注**接口的权限意图，评审时一目了然 | 配置被人误删/改错，接口变公开 |
| **L3 身份认证** | JWT 验证 + Refresh Token **静默刷新**，获取当前用户身份 | 接口里拿不到 `current_user`，也无法做任何用户相关的操作 |

### 9.4 三层独立 + 互相 backup

```
                        L1 中间件配置
                     (配置驱动，全局兜底)
                           │
                    ┌──────┴──────┐
                    │             │
              L1 放行 → 进入路由  L1 拦截 → 403
                    │
            ┌───────┴───────┐
            │               │
       L3 verify_auth()    L3 拦截 → 401
      (JWT验证+静默刷新)
            │
    ┌───────┴───────┐
    │               │
  L2 @RequireRoot  L2 拦截 → 403
   (代码声明加固)
    │
    ▼
  执行业务逻辑 → 记录日志 → 返回
```

即使管理员在面板误操作把某管理路由的 `allowed_roles` 改成了 `["user"]`（或清空了配置），因为代码里还有：

```python
@RequireRoot()
async def handler(current_user=Depends(verify_auth())):
```

- `@RequireRoot()` 会拦截非 root 用户（L2 层兜底）
- `Depends(verify_auth())` 确保 JWT 有效（L3 层兜底）

反过来，如果装饰器被误删了，中间件的路由配置依然能拦截非 root 用户（L1 层兜底）。

> **三层互不依赖，但任何一层通过都不足以放行非授权用户——必须三层全部通过才能执行业务逻辑。**

### 9.5 不同场景的推荐组合

| 场景 | L1 中间件配置 | L2 代码声明 | L3 依赖注入 |
|---|---|---|---|
| 完全公开接口（登录、注册） | `require_auth: false` | 无 | 无 |
| 仅需登录的普通接口 | `_auto: true`（默认放行） | 无 | `Depends(verify_auth())` |
| 普通用户 + 管理员角色区分 | `allowed_roles: ["admin"]` | 无 | `Depends(verify_auth())` |
| root 专属管理接口 | `allowed_roles: ["root"]` | `@RequireRoot()` | `Depends(verify_auth())` |
| 特定用户才能访问 | `allowed_users: [1, 5]` | 无 | `Depends(verify_auth())` |

## 十、配置生命周期总结

一条 API 路由的权限配置从无到有、从松到严的完整过程：

```
1. 开发新接口
   └→ 启动后自动注册到 system_api_route_permission
      └→ _auto: true，中间件放行

2. 测试环境中
   └→ 接口正常工作（_auto 放行），但依赖注入已生效
      └→ Depends(verify_auth()) 确保登录
      └→ @RequireRoot() 确保 root

3. 管理员通过前端面板配置权限
   └→ PUT /system/permission/routes-permission/{path}
      └→ _auto 被移除
      └→ allowed_roles / allowed_users 生效

4. 生产环境中
   └→ 三层同时工作：
      L1 中间件检查配置 → L3 verify_auth() → L2 @RequireRoot()
```
