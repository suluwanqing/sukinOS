# 后端模块详解

## 一、API 路由模块

### 1.1 路由注册总览

在 `exit.py:init_config()` 中注册所有路由：

```python
app.include_router(userInfo)      # /user/*
app.include_router(socketApi)     # /ws/*
app.include_router(sukinOs)       # /sukinos/*
app.include_router(locationSystem) # /system/*
```

### 1.2 用户模块 (`Api/User/userMain.py`)

**前缀**: `/user`

| 路由 | 方法 | 说明 | 认证 |
|---|---|---|---|
| `/user/status/VerificationCode/get` | GET | 获取验证码 | 无 |
| `/user/status/VerificationCode/judge` | POST | 校验验证码 | 无 |
| `/user/status/token` | POST | 登录（密码/验证码） | 无 |
| `/user/status/refresh/token` | POST | 刷新 Token | Refresh Token |
| `/user/status/judge/access_token` | GET | 校验 Token 有效性 | 任意 Token |
| `/user/status/change` | POST | 修改在线状态 | 需要登录 |
| `/user/status/logout` | POST | 注销（清除 Cookie） | 无 |
| `/user/status/user_info_public` | GET | 获取用户公开信息 | 无 |
| `/user/status/profile/update` | POST | 更新个人资料 | 需要登录 |
| `/user/status/password/update` | POST | 修改密码 | 需要登录 |

**关键实现细节**:
- **登录** (`login_for_access_token`): 支持 `account`（账号密码）和 `code`（验证码）两种模式。验证码登录时如果账号不存在，自动注册新账号。
- **密码修改** (`update_user_password`): 需要验证码校验 → 密码哈希 → 签发新 Token → 删除旧验证码缓存。
- **Token 刷新** (`refresh_access_token`): 使用 Refresh Token 签发新 Token 对，支持"记住我"状态继承。

| `/system/usermanage/users` | GET | 用户列表（分页+搜索+过滤） |
| `/system/usermanage/users/{id}` | GET | 用户详情（含周日志） |
| `/system/usermanage/users/{id}/status` | PUT | 启用/禁用用户 |
| `/system/usermanage/users/{id}/info` | PUT | 管理员修改用户信息 |

| 路由 | 方法 | 说明 | 认证 |
|---|---|---|---|
| `/sukinos/app/upload` | POST | 上传 APP | 需要登录 |
| `/sukinos/app/update` | POST | 更新 APP | 需要登录 |
| `/sukinos/app/appList` | GET | 公开 APP 列表 | 需要登录 |
| `/sukinos/app/appList/authorized` | GET | 用户有权限的 APP 列表 | 需要登录 |
| `/sukinos/app/checkUpdates` | POST | 检查 APP 更新 | 需要登录 |
| `/sukinos/app/searchApp` | GET | 搜索 APP | 需要登录 |
| `/sukinos/app/myUpload` | GET | 我的上传记录 | 需要登录 |
| `/sukinos/app/delete` | POST | 删除自己的 APP | 需要登录 |
| `/sukinos/localdev/*` | - | 本地开发同步 | 需要登录 |

**上传流程**:
1. 检查文件大小（≤10MB）
2. 检查磁盘空间（保留 20GB 安全阈值）
3. 生成安全文件名（APP名_版本号_防重复编码）
4. 使用存储抽象层保存文件（目前本地磁盘）
5. 写入数据库，状态为 `under_review`
6. 管理员审核后变为 `active` 或 `disabled`

## 二、中间件层

### 2.1 RoutePermissionMiddleware

**文件**: `Middleware/routePermission.py`

全局路由权限检查的核心实现。详见[系统架构](./2-系统架构.md)的中间件详解部分。

**白名单配置结构**:
```json
{
  "publicPaths": [],
  "publicPrefixes": ["/static", "/resource", "/user/status/"],
  "selfPrefixes": ["/system/permission/"]
}
```

### 2.2 StaticAuthMiddleware

**文件**: `Middleware/staticAuth.py`

使用**处理器映射模式**支持多个静态资源挂载点：

```python
AUTH_HANDLERS = {
    APP_FILE_MOUNT: handle_sukinos_app_auth,    # "/resource/sukinos/app/download"
    RESOURCE_USER_MOUNT_PATH: handle_public_auth,  # "/media/user"
}
```

### 2.3 RequestLogMiddleware

**文件**: `Middleware/requestLog.py`

纯 ASGI 实现，避免 `BaseHTTPMiddleware` 的 `TaskGroup` 并发错误。

**日志过滤配置**:
```json
{
  "rules": [
    {"method": "GET", "path": "/system/status", "enabled": false}
  ],
  "logRequestBody": true,
  "logResponseBody": true
}
```

## 四、数据模型层 (`Model/`)

| 类别 | 说明 |
|---|---|
| `DbModel/` | SQLAlchemy ORM 模型 |
| `RequestModel/` | Pydantic 请求参数验证模型 |
| `ResponseModel/` | Pydantic 响应体序列化模型 |

所有请求/响应模型使用 Pydantic v2 验证，响应通过 `api_response()` 统一包装并自动转驼峰。

## 五、异步任务 (`CeleryWork/`)

| 配置项 | 值 |
|---|---|
| Broker | Redis (db 0) |
| Backend | Redis (db 1) |
| 任务模块 | `CeleryWork.VerificationCode` |
| 并发池 | threads, 2 workers |

**当前任务**:
- 验证码邮件发送（异步，非阻塞）

## 六、Redis 缓存

| 用途 | DB |
|---|---|
| Celery Broker | 0 |
| Celery Backend | 1 |
| 验证码缓存 | 5 |

Redis 主要用于验证码的临时存储与限流管理。
