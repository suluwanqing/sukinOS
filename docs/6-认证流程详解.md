# 认证流程详解

## 一、认证机制概述

SukinOS 采用**双 Token JWT 认证机制**，结合 Cookie 存储、静默刷新、多模式登录，构建完整的用户认证体系。

### 核心组件

| 组件 | 文件 | 说明 |
|---|---|---|
| Token 生成 | `Api/Common/Auth.py` | 签发 Access/Refresh Token |
| Token 验证 | `Api/Common/Auth.py:verify_auth()` | FastAPI 依赖注入 |
| Cookie 管理 | 各登录/刷新接口 | 设置/清除 Cookie |
| 密码哈希 | `passlib.context.CryptContext` (bcrypt) | 128位哈希 |
| 验证码 | `Func/system/verificationCodeManager.py` | Redis 缓存验证码 |

## 二、Token 机制

### 2.1 Access Token

| 属性 | 值 |
|---|---|
| 有效时长 | 30 分钟（ACCESS_TOKEN_EXPIRE_MINUTES） |
| Cookie Key | `access_token` |
| Payload Type | `access` |
| 用途 | 所有业务请求的身份凭证 |

### 2.2 Refresh Token

| 属性 | 值 |
|---|---|
| 有效时长 | 7 天（LONG_REFRESH_TOKEN_EXPIRE） |
| Cookie Key | `refresh_token` |
| Payload Type | `refresh` |
| 用途 | Access Token 过期时静默刷新 |

### 2.3 Payload 结构

```json
{
  "sub": "user_account",
  "type": "access",
  "remember": true,
  "exp": 1700000000
}
```

**注意**：Token 内部的有效期是固定的（30min/7天），"记住我"功能通过 Cookie 的 `max_age` 实现：
- **记住我**：`max_age=有效期秒数`，浏览器持久化存储
- **不记住我**：`max_age=None`，会话 Cookie（关闭浏览器即消失）

## 三、完整登录流程

### 3.1 密码登录

```
用户提交账号密码
  │
  ▼
POST /user/status/token
  │ type=account
  │
  ▼
查询用户 (or_ account/email)
  │
  ├─ 用户不存在 → 返回 403
  │
  ├─ 用户被禁用(is_active=false) → 返回 403
  │
  └─ 用户存在
      │
      ▼
      verify_password(明文密码, hash_ps)
      │
      ├─ 不匹配 → 返回 403
      │
      └─ 匹配
          │
          ▼
          签发 Access Token + Refresh Token
          │
          ▼
          设置 Cookie:
          - access_token: "Bearer <jwt>" max_age=取决于remember
          - refresh_token: "Bearer <jwt>" max_age=取决于remember
          │
          ▼
          返回 Token + User 信息
```

### 3.2 验证码登录

```
用户提交验证码
  │
  ▼
POST /user/status/token
  │ type=code
  │
  ▼
校验验证码 (Redis 中匹配)
  │
  ├─ 校验失败 → 返回 403
  │
  ├─ 用户已存在但被禁用 → 返回 403
  │
  ├─ 用户已存在且正常 → 签发 Token
  │
  └─ 用户不存在 → 自动注册新账号
      │
      ▼
      init_account(account, tempPassword)
      │ 创建 User + UserExpand + UserTimeBehavior
      │
      ▼
      签发 Token + 设置 Cookie
```

### 3.3 验证码获取与校验流程

```
前端请求验证码
  │ GET /user/status/VerificationCode/get
  │ params: {account, message, type, seed}
  ▼
后端：
  1. 生成随机验证码
  2. Redis 存储: VerificationCode:{type}:{account} → {seed, code, id}
  3. 设置限流: VerificationCodeLimit:{type}:{account} → TTL 60s
  4. 发送邮件
  5. 返回 {id, seed, limit}

前端提交验证码
  │ POST /user/status/VerificationCode/judge
  │ params: {id, account, code, seed, type}
  ▼
后端：
  1. 从 Redis 取出记录
  2. 比对 id + seed + code 三者完全匹配
  3. 校验通过 → 删除 Redis 缓存
```

## 四、静默刷新机制

SukinOS 实现了**双处静默刷新**，分别在不同的层级处理：

### 4.1 路由层刷新 (`verify_auth`)

**文件**: `Api/Common/Auth.py:63-218`

```
接口请求
  │
  ▼
Depends(verify_auth())
  │
  ├─ access_token Cookie 存在且有效
  │   → 解析 JWT → 获取用户 → 返回
  │
  ├─ access_token 过期/无效
  │   ├─ refresh_token 不存在 → 401 会话过期
  │   │
  │   └─ refresh_token 存在
  │       ├─ refresh_token 过期 → 401 长期登录失效
  │       │
  │       └─ refresh_token 有效
  │           ├─ 校验 type=refresh
  │           ├─ 查询用户是否存在
  │           ├─ 签发新 Access Token + Refresh Token
  │           ├─ 通过 response.set_cookie() 写入新 Cookie
  │           └─ 标记 user._refreshed = true
  │
  └─ 返回 User 对象，继续执行路由
```

### 4.2 中间件层刷新 (`StaticAuthMiddleware.authenticate_user`)

**文件**: `Middleware/staticAuth.py:114-144`

静态资源访问时的身份验证逻辑：
```
1. 优先使用 access_token Cookie
2. 失败则尝试 refresh_token Cookie
3. 也支持 Authorization Header Bearer Token
4. 刷新成功时返回 (user, new_tokens)
5. 在 _inject_refresh_cookies() 中写入响应 Cookie
```

### 4.3 显式刷新接口

```
POST /user/status/refresh/token
  Cookie: refresh_token=...
  │
  ▼
Depends(verify_refresh_token_dep)
  │ 验证 Refresh Token 合法性
  │ 提取 remember 状态
  │
  ▼
签发新 Token 对 → 更新 Cookie → 返回
```

## 五、密码管理

### 5.1 密码修改

```
POST /user/status/password/update
  │ body: {new_password, code, code_id}
  │ params: {seed}
  │ Cookie: access_token
  │
  ▼
1. validate: 验证码校验
2. validate: 新密码长度 ≥ 6
3. hash: pwd_context.hash(new_password)
4. commit: 更新 hash_ps
5. refresh: 签发新 Token 对
6. cleanup: 删除 Redis 验证码缓存
7. return: 新 Token + User
```

**安全要点**：
- 修改密码需要验证码二次验证
- 修改后原 Token 仍有效（直到过期），新 Token 立即签发
- 前端可提示用户其他设备将因 Token 过期而需要重新登录

### 5.2 密码存储

```python
# 哈希
hash_ps = pwd_context.hash(password)
# pwd_context = CryptContext(schemes=["bcrypt"])

# 验证
verify_password(password, db.hash_ps)
# 使用 bcrypt 算法，自动处理 salt
```

## 六、登出流程

```
POST /user/status/logout
  │
  ▼
response.delete_cookie("access_token")
response.delete_cookie("refresh_token", path="/")
  │
  ▼
返回成功
```

**注意**：
- refresh_token 需要指定 `path="/"` 删除，因为签发时指定了根路径
- 登出不会使 Token 失效（无黑名单机制），但 Cookie 被清除后浏览器不再发送
- 这是无状态 JWT 的标准做法

## 八、Cookie 配置

```python
response.set_cookie(
    key="access_token",
    value=f"Bearer {token}",
    max_age=cookie_max_age,  # 记住我时 = 有效期，否则 = None
    httponly=True,           # 禁止 JS 读取（防 XSS）
    secure=True,             # 仅 HTTPS
    samesite="lax",          # CSRF 防护
    path="/"                 # 全站生效
)
```

**安全策略**：
- `httponly`：防止 XSS 窃取 Token
- `secure`：确保仅通过 HTTPS 传输
- `samesite=lax`：防止 CSRF 攻击
- 前缀 `Bearer`：遵循 RFC 6750 标准

### 9.1 限流

```python
# 每个账号每种类型 60秒内只能发一次
```

```python
# 验证使用 id + seed + code 三重校验
```
