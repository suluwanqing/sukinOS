# SukinOS 进程通信架构

## 一、架构总览

SukinOS 进程系统采用 **4 层通信架构**，以 `CommHub` 为中心枢纽，连接 UI 渲染层、内核逻辑层与 Worker 沙箱层。

```
┌─────────────────────────────────────────────────────────────────────┐
│                         UI 渲染层                                    │
│  useKernel (系统级订阅)   │   useProcessBridge (App 级订阅)            │
│  subscribeSystem()        │   subscribeApp(pid, cb)                  │
│                           │   dispatch(pid, action)                   │
│                           │   publish(topic, payload)                 │
│                           │   subscribe(topic, cb)                    │
└──────────┬────────────────┼────────────────┬────────────────────────┘
           │                │                │
           │ sys_change      │ STATE           │ topic
           ▼                ▼                ▼
┌─────────────────────────────────────────────────────────────────────┐
│                      CommHub 通信枢纽                                │
│  ┌──────────────┐ ┌──────────────┐ ┌────────────────────────────┐   │
│  │ eventBus     │ │ subscribers  │ │ topicSubscribers            │   │
│  │ sys_change   │ │ pid → Set(cb)│ │ topic → Set(cb)            │   │
│  └──────────────┘ └──────────────┘ │ processSubscriptions        │   │
│  ┌──────────────┐ ┌──────────────┐ │ pid → Map(topic → unsub)    │   │
│  │ stateCache   │ │ handleMsg()  │ └────────────────────────────┘   │
│  │ pid → state  │ │              │ ┌──────────────┐                │
│  └──────────────┘ └──────────────┘ │ sendToWorker │                │
│                                    └──────────────┘                │
└────────────────────────────────────┬────────────────────────────────┘
                                     │ postMessage
                                     ▼
┌─────────────────────────────────────────────────────────────────────┐
│                   Worker / VirtualWorker 沙箱                        │
│  onmessage:                                                         │
│    INIT / RESTORE / UI_ACTION / APP_INTERACT                        │
│  postMessage:                                                       │
│    STATE_UPDATE / SAVE_STATE / PUBLISH_TOPIC                        │
│    SUBSCRIBE_TOPIC / UNSUBSCRIBE_TOPIC                             │
└─────────────────────────────────────────────────────────────────────┘
```

## 二、5 大通信通道

### 通道 1：Kernel → UI（系统变更事件）

**机制**: `emitChange(detail)` → `CommHub.emitChange()` → `EventTarget.dispatchEvent('sys_change')` → `subscribeSystem(cb)`

**入口**: `kernelParts/messaging.js` → `commHub.js:20-22`
**订阅方**: `usekernel.js` 中的 `applyDelta(detail)` 函数

**消息格式**:
```js
emitChange({ type: 'APP_STATUS', pid, status: 'RUNNING' })
```

####  固定 sys_change 事件类型

| type | 触发场景 | detail 结构 | 说明 |
|------|---------|-------------|------|
| **`APP_STATUS`** | 应用启动/休眠/恢复 | `{ type, pid, status }` | status: `RUNNING` / `HIBERNATED` / `INSTALLED` |
| **`APP_META`** | 应用 custom 配置变更 | `{ type, pid, resourceId, meta }` | 元信息更新 |
| **`APP_REGISTRY`** | 安装/卸载/更新应用 | `{ type }` | 注册表变化，触发全量刷新 |
| **`APP_STATE_CLEARED`** | 清除应用持久化状态 | `{ type, pid }` | 状态被主动清空 |
| **`APP_RESET`** | 强制重置应用 | `{ type, pid }` | 杀进程 + 清状态 + 标记 INSTALLED |
| **`APP_RESTART`** | 重启应用 | `{ type, pid }` | reStartApp / forceReStartApp |

> **固定规则**: `type` 字段是固定枚举值，`applyDelta` 函数对已知类型做精确更新，对未知类型触发全量刷新。

---

### 通道 2：UI → Worker（指令下发）

**机制**: UI 调用 `dispatch(pid, action)` → `kernelParts/messaging.js:dispatch()` → 判断是否为内核调用 → `postMessage` 转发

**入口**: `useProcessBridge.jsx` 第 48 行
**路由**: `messaging.js:dispatch()`

#### 消息路由逻辑

```
dispatch(pid, action)
  ├── action.type === 'KERNEL_CALL' → 系统调用路由
  │     ├── 系统App → systemSwitch()
  │     │     └── UPLOAD_RESOURCE: 上传资源
  │     └── 非系统App → notSystemSwitch()
  └── 其他 type → worker.postMessage({ type: 'UI_ACTION', payload: action })
```

####  固定入站消息类型（Worker 接收）

| type | 方向 | 说明 | 是否固定 |
|------|------|------|---------|
| **`INIT`** | Kernel → Worker | 初始化 Worker，执行 `init()` 函数，广播初始状态 | 🔒 固定 |
| **`RESTORE`** | Kernel → Worker | 恢复休眠状态，`payload` 为保存的 state 快照 | 🔒 固定 |
| **`UI_ACTION`** | Kernel → Worker | UI 层发来的用户交互指令，`payload` 为用户 action 对象 | 🔒 固定类型名，payload 可扩展 |
| **`APP_INTERACT`** | Kernel → Worker | App 间交互消息，`payload` 为交互信息 | 🔒 固定类型名，payload 可扩展 |

####  固定 KERNEL_CALL 方法名

| method | 说明 | 调用者 |
|--------|------|--------|
| **`UPLOAD_RESOURCE`** | 上传资源到云端 | 系统App |

---

### 通道 3：Worker → Kernel → UI（状态上报）

**机制**: Worker 内部 `broadcast()` → `postMessage({ type: 'STATE_UPDATE' })` → Kernel 侧 `worker.onmessage` → `commHub.handleMsg()` → `notify()` → UI 订阅者回调

####  固定出站消息类型（Worker 发送）

| type | 说明 | CommHub 处理逻辑 |
|------|------|------------------|
| **`STATE_UPDATE`** | 状态变更广播 | 写入 `stateCache` + 通知所有 `subscribers.get(pid)` |
| **`SAVE_STATE`** | 持久化状态请求 | 调用 `onSaveState(pid, payload)` 写入 IndexedDB |

#### 状态通知格式

```js
// CommHub.notify 发送给 UI 订阅者的格式
{ type: 'STATE', payload: { ...userState, config: SYS_CONFIG } }
```

####  固定状态更新链

```
Worker 内部:
  dispatch(action)
    → reducer(state, action) = nextState
    → _state = nextState
    → broadcast()           // 固定: 发送 STATE_UPDATE
    → save()                // 固定: 发送 SAVE_STATE
```

---

### 通道 4：跨 App 主题通信（Pub/Sub）

**机制**: 基于 `CommHub.topicSubscribers` 的发布/订阅模式，支持 App 跨沙箱通信。

####  固定主题通信消息类型

| type | 方向 | 说明 |
|------|------|------|
| **`PUBLISH_TOPIC`** | Worker → CommHub | App 发布主题消息。payload: `{ topic, data }` |
| **`SUBSCRIBE_TOPIC`** | Worker → CommHub | App 订阅主题。payload: `{ topic }`。CommHub 代理订阅后，将收到的消息转发回 Worker |
| **`UNSUBSCRIBE_TOPIC`** | Worker → CommHub | App 取消订阅。payload: `{ topic }` |
| **`TOPIC_MESSAGE`** | CommHub → Worker | 系统向已订阅的 Worker 推送主题消息。payload: `{ topic, data }` |

#### 通信流程

```
App A (Worker)                    CommHub                     App B (Worker)
     │                                │                              │
     │── PUBLISH_TOPIC ──────────────→│                              │
     │   { topic: 'chat', data }     │                              │
     │                                │── TOPIC_MESSAGE ──────────→ │
     │                                │   { topic: 'chat', data }  │
     │                                │                              │
     │                                │←── SUBSCRIBE_TOPIC ──────── │
     │                                │   { topic: 'chat' }         │
     │                                │                              │
     │                                │── TOPIC_MESSAGE ──────────→ │
     │                                │   { topic: 'chat', data }  │
```

####  固定规则

- `topic` 名称：**不固定**，由 App 自行定义（如 `'chat'`、`'file-sync'` 等）
- `data` 格式：**不固定**，由发布方和订阅方协商
- 进程销毁时，`clearProcessSubscriptions(pid)` 自动清理该进程的所有主题订阅

---

### 通道 5：App 间交互（点对点）

**机制**: `appIntereact()` → 直接向目标 Worker 的 `postMessage` 发送 `APP_INTERACT` 消息

**入口**: `messaging.js:appIntereact(kernel, { process, interactInfo })`

```js
// 调用方式
kernel.appIntereact({
  process: targetWorker,      // 目标 Worker 实例
  interactInfo: { type: 'xxx', payload: {...} }  // 交互数据
})
```

Worker 内部对 `APP_INTERACT` 的处理与 `UI_ACTION` 完全相同：均调用 `dispatch(payload)` 进入用户 reducer。

## 三、固定协议 vs 可扩展协议

###  固定协议（系统保留，不可更改）

| 分类 | 名称 | 说明 |
|------|------|------|
| **事件名** | `sys_change` | EventTarget 固定事件名，系统变更通知 |
| **sys_change types** | `APP_STATUS` / `APP_META` / `APP_REGISTRY` / `APP_STATE_CLEARED` / `APP_RESET` / `APP_RESTART` | 6 种固定系统事件类型 |
| **Worker 入站** | `INIT` / `RESTORE` / `UI_ACTION` / `APP_INTERACT` | 4 种固定入站消息类型 |
| **Worker 出站** | `STATE_UPDATE` / `SAVE_STATE` | 2 种固定出站消息类型 |
| **主题协议** | `PUBLISH_TOPIC` / `SUBSCRIBE_TOPIC` / `UNSUBSCRIBE_TOPIC` / `TOPIC_MESSAGE` | 4 种固定主题通信消息类型 |
| **状态通知 type** | `STATE` | CommHub.notify 发给 UI 订阅者的固定 type |
| **内核调用 type** | `KERNEL_CALL` | dispatch 中识别内核调用的固定 action.type |
| **内核方法** | `UPLOAD_RESOURCE` | KERNEL_CALL 中的固定 method 名 |
| **内核调用转发** | Worker 收到后仍以 `UI_ACTION` 转回 | systemSwitch 内部固定行为 |

###  可扩展协议（App 开发者自定义）

| 分类 | 说明 | 示例 |
|------|------|------|
| **UI_ACTION payload** | Worker 收到的 action.type 和 action.payload 完全由 App 开发者定义 | `{ type: 'INCREMENT' }`, `{ type: 'NAVIGATE', payload: '/settings' }` |
| **APP_INTERACT payload** | App 间交互的数据格式由交互双方约定 | `{ type: 'SEND_FILE', payload: { url, name } }` |
| **Topic 名称** | 跨 App 通信的主题名由开发者自定义 | `'chat'`, `'clipboard'`, `'theme-change'` |
| **Topic data** | 主题消息的 payload 格式由发布/订阅方约定 | 任意结构 |

## 四、完整消息流转图

### 4.1 App 启动流程

```
UI: useProcessBridge.startProcess(pid)
  └→ kernel.startProcess({ pid, resourceId, interactInfo })
       └→ 创建 Worker / VirtualWorker
          Worker.onmessage = (e) → commHub.handleMsg(pid, e.data)

  Kernel → Worker: { type: 'INIT' }
    └→ Worker: _state = initialState; init(dispatch, _state)
       └→ broadcast() → { type: 'STATE_UPDATE', payload: state+config }

  Worker → CommHub: STATE_UPDATE
    └→ stateCache.set(pid, payload)
    └→ notify(pid, 'STATE', payload)
       └→ UI: useProcessBridge.setState(payload)

  Kernel → CommHub: emitChange({ type: 'APP_STATUS', pid, status: 'RUNNING' })
    └→ UI: useKernel.applyDelta → 更新应用列表
```

### 4.2 UI 交互流程

```
UI: dispatch({ type: 'INCREMENT' })
  └→ kernel.dispatch(pid, { type: 'INCREMENT' })
    └→ worker.postMessage({ type: 'UI_ACTION', payload: { type: 'INCREMENT' } })

  Worker: onmessage → dispatch(payload)
    └→ reducer(state, { type: 'INCREMENT' }) → nextState
       └→ broadcast() → STATE_UPDATE → ... → UI 更新
```

### 4.3 休眠与恢复流程

```
// 休眠
kernel.hibernate(pid)
  └→ 标记 HIBERNATED (Worker 不终止，保持热状态)
  └→ emitChange({ type: 'APP_STATUS', pid, status: 'HIBERNATED' })

// 恢复
kernel.startProcess({ pid, ... })
  └→ 发送 { type: 'RESTORE', payload: savedState } 到已存在的 Worker
  └→ Worker: _state = payload; broadcast()
```

### 4.4 跨 App 通信流程

```
// App A 发布
SDK.Topic.publish('clipboard', { text: 'hello' })
  └→ worker.postMessage({ type: 'PUBLISH_TOPIC', payload: { topic: 'clipboard', data: { text: 'hello' } } })
    └→ CommHub.publish('clipboard', { text: 'hello' })
      └→ 所有订阅 'clipboard' 的回调执行

// App B 订阅
SDK.Topic.subscribe('clipboard', (data) => { ... })
  └→ worker.postMessage({ type: 'SUBSCRIBE_TOPIC', payload: { topic: 'clipboard' } })
    └→ CommHub 代理订阅，收到消息后转发 TOPIC_MESSAGE 给 App B 的 Worker
```

## 六、VirtualWorker（虚拟沙箱）

`lifecycle.js` 中的 `VirtualWorker` 类提供了一个**无需真实 Web Worker 线程**的轻量沙箱方案。它通过 `with(sandbox) + Proxy + Shared Iframe` 模拟 Worker 环境。

**与真实 Worker 的通信差异**:
- `VirtualWorker.postMessage()` 内部通过 `requestAnimationFrame` 节流 `STATE_UPDATE` 消息
- 消息不经过序列化，直接传递对象引用
- 拦截接管所有定时器以便在 `terminate()` 时强制清理
- 对外暴露的 `onmessage` / `postMessage` 接口与真实 Worker 完全一致
